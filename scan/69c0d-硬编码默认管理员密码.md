## 1. 总结
- **漏洞类型**：硬编码凭证（CWE-798）
- **标记位置**：`app/models/db.py:254`
- **漏洞描述**：数据库初始化函数 `init_db()` 创建了一个默认超级管理员账户，其密码为硬编码的明文密码 `"admin888"`。初始化过程中该密码被输出到标准输出，因此任何能够访问代码或启动日志的人都可以轻易获取管理员账户权限。

## 2. 分析逻辑

### 步骤 1：检查标记的汇点（sink）`app/models/db.py:251-259`
```python
if not admin_exists:
    # 不存在，创建默认超级管理员
    salt = secrets.token_bytes(16)
    password_hash = _hash_password("admin888", salt)
    conn.execute(
        "INSERT INTO admins (username, password_hash, salt, role_id) VALUES (?, ?, ?, ?)",
        ("admin", password_hash, salt.hex(), 2)
    )
    print("默认超级管理员创建成功！用户名: admin, 密码: admin888")
```
密码 `"admin888"` 是直接传入 `_hash_password` 函数的字符串字面量。这是一个硬编码凭证。`print` 语句进一步将密码泄露到标准输出。接下来需要检查这段代码的调用位置以及密码是否会被覆盖。

## 3. 结论
**真实漏洞**

**关键证据：**
- 明文密码 `"admin888"` 作为字符串字面量硬编码在 `app/models/db.py:254` 中。
- `init_db()` 在应用程序启动时无条件调用（`app.py:61`），如果默认管理员不存在，则使用此密码创建。
- 项目中没有环境变量加载或密钥管理器集成（在 `app/` 下搜索 `os.getenv`/`environ` 未返回任何结果）。
- `app/models/db.py` 未被版本控制排除（未在 `.gitignore` 中）。
- 密码在 `app/models/db.py:259` 中输出到标准输出，导致在启动日志中泄露。

## 4. 修复建议
- **完全移除硬编码密码**。不要在源代码中嵌入 `"admin888"` 或任何默认密码。
- **实现首次运行设置流程**。在应用程序首次启动时，要求管理员通过安全设置页面或 CLI 提示设置一个强密码，并仅存储最终的哈希值。
- **如果必须以编程方式创建默认账户**，生成一个密码学安全的随机强密码（例如 `secrets.token_urlsafe(32)`），进行哈希处理后，通过以下方式之一处理：
  - 在安全环境下**一次性显示**（例如在设置向导中，而非 stdout/日志中），或者
  - 写入一个具有受限权限的一次性文件，或者
  - 通过安全的带外通道发送。
- **切勿记录或打印明文密码**。移除或替换 `app/models/db.py:259` 中的 `print` 语句。
- **考虑将必要的初始化秘密从环境变量加载**（例如 `os.getenv('ADMIN_INITIAL_PASSWORD')`）作为短期缓解措施，但对于默认凭据，优先使用首次运行设置而非环境变量。
